# Atlas Vendor + Guest Portal Strategy

## Why This Direction Is Strong

This is a high-leverage product move for REVEL:

- Couples get faster decisions and fewer surprises.
- Vendors get operational clarity and fewer coordination gaps.
- Venues get cleaner execution and fewer policy violations.
- Guests get utility (not just info), which creates silent brand exposure.

Efficiency-first fit:
- Reduces WhatsApp churn.
- Reduces duplicate planner/vendor questions.
- Reduces day-of confusion and avoidable penalties.

## Product Model

Use a 3-surface system on one shared intelligence core.

1. Client Surface (Couples)
- AI concierge
- Budget guidance
- Venue fit recommendations
- Timeline confidence dashboard

2. Vendor Surface (Private)
- Access only to weddings they are assigned to
- Venue logistics and restrictions
- Program timeline + handoff markers
- Role-specific run sheets (DJ, decor, photo/video, catering, planner)
- Issue escalation and change log

3. Guest Surface (Lightweight)
- Event schedule and location navigation
- Parking and arrival instructions
- Cultural ceremony primer (short and elegant)
- RSVP reminders and simple updates
- Optional branded utility pages that softly advertise REVEL capability

## Permission Model (Minimum)

- Couple/Admin: full visibility and approvals
- Planner: full timeline + vendor coordination + notes
- Vendor: only assigned event and role-scoped tabs
- Guest: read-only, curated event experience

Authentication approach:
- Magic links for guests and simple role access
- Vendor seat invite + event-bound session
- Admin audit logs for edits to critical event data

## Vendor-Only Workspace MVP (Phase 1)

Ship this first because it has immediate weekend-level ROI.

### Core Modules

1. Venue Reality
- Indoor/outdoor route plans
- Sound/power notes
- Restrictions (open flame, CO2, haze, curfew, load-in)
- Freight and access notes (dock, elevator, door width)

2. Event Runbook
- Locked timeline version
- Cue list by role
- "No-silence" continuity markers for entertainment
- Rain fallback branch (for baraat-style transitions)

3. Coordination Feed
- Last-minute changes with timestamps
- Owner + assignee per change
- Status: pending, acknowledged, executed

4. Risk Flags
- Capacity pressure
- Weather fallback risk
- Policy conflict risk
- Setup time compression risk

## Guest Portal MVP (Phase 2)

- Mobile-first event hub with itinerary and FAQs
- Arrival timing prompts to reduce late guest drag on program
- Ceremony etiquette prompts (very short, warm tone)
- CTA to explore REVEL planning tools post-event (soft conversion)

## Data Contract for Venue Intelligence

Every venue field should include trust metadata:

- value
- source_type (site call, contract, venue email, on-site note)
- source_link or evidence reference
- verified_by
- verified_on
- confidence_score
- expiry_or_review_date

## DoubleTree-Type Nuance Capture (Example Categories)

- fire/open flame policy
- indoor baraat path feasibility
- welcome bag handling policy
- mgmt stability/turnover history
- bar package structure and practical constraints
- where DJ/speakers are acceptable indoors
- attrium/lobby partition notes that affect crowd flow and sound

## Efficiency KPIs (Vendor + Client)

- reduce pre-event coordination messages per event by 30%+
- reduce timeline clarification calls by 40%+
- reduce day-of coordination escalations by 25%+
- reduce avoidable policy violations to near zero
- reduce venue research time for similar properties by 50%+
